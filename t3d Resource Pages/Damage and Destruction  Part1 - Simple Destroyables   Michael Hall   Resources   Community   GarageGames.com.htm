Damage and Destruction: Part1 - Simple Destroyables

by Michael Hall · 04/05/2011 (2:10 pm) · 22 comments

All ShapeBase objects in Torque can recieve damage and be destroyed. However, only the Player class takes advantage of this built in functionality in the stock scripts. Objects such as Vehicles, StaticShapes, RigidShapes, etc can easily use the damageState system as well.

This is (according to current plan but adjusted by available time) going to be a series of Tutorials/Guides of increasing complexity starting with this basic destroyble and over time showing more advanced concepts such as enabling onDamage type impact effects (per object) as well as sounds, wreckage shapes, animated damage sequences, etc.

The example I am about to describe is not the physics enabled destruction allowed by Torque 3D's Physics Shapes - there is an already a guide showing how to use Physics Shapes by Joe Espinoza in this forum thread.

Lets start by making the BouncingBoulder RigidShape example in the Full Template destroyable. It will be able to recieve x amount of damage (set in the datablock). Once the damage recieved exceeds that threshold we will simulate the BouncingBoulder being destroyed by using an Explosion object and associated Particle effects, and in the process deleting the boulder.dts shape from the scene.

These are the "damage control" properties we will make use of:
  • maxDamage
  • disabledLevel
  • destroyedLevel
  • renderWhenDestroyed
  • Explosion
  • debrisShapeName EDIT: not used for this Resource
  • debris EDIT: not used for this Resource

EDIT: There is a bug in Torque 3D having to do with the ShapeBase blowup feature. This Resource had simply overlooked the feature. To keep this a 'simple' Resource I will comment below at the location were the bug is encountered, but will continue to not make use of this for this Resource. Adding the fix, an example object, and illustrating how to properly make use of the blowup-into-chunks functionality will come as another Resource

function BouncingBoulder::damage(%data, %obj, %sourceObject, %position, %amount, %damageType)
{
   //echo("BouncingBoulder::damage("@ %data.getName() @", "@ %obj @", "@ sourceObject @", "@ %position @","@ %amount @", "@ %damageType@")");
   
   // There's no need to calculate damage for an object that's already destroyed,
   if (%obj.isDestroyed())
   {
      //echo("object already destroyed, returning");
      return;
   }
         
   // We can check for certain damageTypes and/or scale the damage for 
   // different types before applying damage.  Another option here would be to
   // simply return if the %damagetype !$= "ExplosionDamage" or whatever type
   // of damage you wish capable of causing destruction.
    %obj.applyDamage(%amount);
}

function BouncingBoulder::onDamage(%data, %obj)
{
   //echo("BouncingBoulder::onDamage("@ %data @", "@ %obj @")");
   
   // Set damage state based on current damage level.  We are comparing amount 
   // of damage sustained to the damage levels described in the datablock and 
   // setting the approppriate damageState
   
   %damage = %obj.getDamageLevel();
   
   if (%damage >= %data.destroyedLevel)
   {
      if (%obj.getDamageState() !$= "Destroyed")
      {
         %obj.setDamageState(Destroyed);
         %obj.setDamageLevel(%data.maxDamage);
      }
   }
   else if(%damage >= %data.disabledLevel)
   {
      if (%obj.getDamageState() !$= "Disabled")
         %obj.setDamageState(Disabled);
   }
   else
   {
      // we're just assuming that the object is still nice and healthy
      if (%obj.getDamageState() !$= "Enabled")
         %obj.setDamageState(Enabled);
   }
}

function BouncingBoulder::onDisabled(%data, %obj, %state)
{
   // nothing for now...
}

function BouncingBoulder::onDestroyed(%data, %obj, %prevState)
{
   //echo("BouncingBoulder::onDestroyed("@ %data @", "@ %obj @", "@ %prevState @")");
    
   // If this is set to false then we delete the object when it is destroyed, 
   // we do so while it is still obscured by the explosion fx  
   if (!%data.renderWhenDestroyed)
      %obj.schedule(200, "delete");
}
Take note that even though these methods are currently set to the datablock namespace they could just as easily be used more universally by writing them for a class namespace.

Save. Start Game. Litter a few BouncingBoulders around the level and go blow them up! Example settings used here (see maxDamage, disabledLevel, and destroyedLevel in the datablock) require about 4-5 shots with the rockelauncher so adjust to taste.

The careful observer will notice that very little actual code had to be written to enable this destructible functionality. Most of the work is handled automatically by the engine simply by adding a few lines of logic and the requisite properties to an object's datablock.

Next Guide (as time permits) will demonstrate how to add "onHit" effects, such as sounds and/or sparks/dustpuffs/etc) to our destroyable object as it is damaged.

Your approach, seems to have the boulder break apart in 'generic' shape...with only one piece of geometry in the file...m'eh, hard to tell the original object is 'tearing apart'...

The 'key' to the art is to have several geometry groupings in the file, each an 'object', with node. The engine takes the 'nodes'/object and does the automagic from there.

It's kinda tricky with the naming of things...Like Static Mesh and TSStatic Shape.

Generic explosions...like the crossbow projectile one...point to a single piece of geometry...thus each 'flung' piece[having only a single node in the DTS] will be replicated according to the explosion datablock.

Now, with debrisShape...these are the nonGeneric pieces, looking like real destruction, what you see takes off flying from where it was exported...

Then there's what is left over after the explosion[more Debris naming...]...so there is a lot to an 'exploding' object!!

Took me a while to sort thru datablocks and what art is needed for what effect...;) For Torque Art...you need to be a "Technical Artist" more than a 'line,shape,form' Artist! A Zbrush million poly figure isn't worth much if it can't get into Torque and doing something interesting...;).

Bug encountered, disclaimer, and goal thereof.


I think that I should clarify in the Resource that this is only a simple example of having an object take damage and be destroyed. The explosion, debris, and particles are simply there to obscure the fact that the original object is being deleted once it's destroyedState is reached.

There is a problem with using a break-into-chunks debris shape. A nasty crashing the game kind of problem if such a shape is used. I had simply overlooked this feature. Couldn't see the forest because of the trees, so to speak, due to explosion fx and it's accompanying debris (which is different from "debris shapes". But there is hope. The bug can be fixed, and object-into-chunks debris can be had for all. Guy Allard posted a bug report with fix to this issue.

Just to note, that currently, the BouncingBoulder datablock is referencing the debrisShapeName and Debris object incorrectly (this is that part I overlooked). But this should not cause any problems (hasn't for me) since this means that the "blowUp" feature simply wont' be used. Debris seen in the preview pic is coming from the explosion itself.
